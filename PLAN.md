## Executive Summary

Phase 1 is complete (251 tests passing). The outstanding blocker is the Drizzle migration
pipeline: `pnpm db:migrate` fails on a fresh install because the `drizzle/meta/` directory
was generated by hand rather than by `drizzle-kit generate`. Running `pnpm db:generate`
fixed the journal, but revealed a secondary problem: `0002_partition_trades.sql` hard-codes
the `2026-04` partition range. On any fresh DB created after April 2026 the first inserted
row would fail with **"no partition of relation found for row"**.

Law's board review identified two findings:

| Severity | Finding |
|---|---|
| **MAJOR** | Hard-coded `2026-04` partitions in `0002_partition_trades.sql` (lines 47, 88) |
| **MINOR** | No `0001_snapshot.json` in `drizzle/meta/` for the manually-registered `0002` journal entry |

Both are addressed in the tasks below.

## Task List (atomic, ordered)

### Task 1 — Delete orphan `0001_initial_schema.sql`

**File:** `drizzle/0001_initial_schema.sql`  
**Action:** Delete the file.  
**Why:** `0000_misty_thaddeus_ross.sql` (generated by `drizzle-kit`) is the semantic
equivalent — all 8 tables, same column definitions, no `IF NOT EXISTS` guards. Keeping both

**Test:** `ls drizzle/` should show `0000_misty_thaddeus_ross.sql` and `0002_partition_trades.sql` only (plus `meta/`).

### Task 2 — Rewrite `0002_partition_trades.sql` with dynamic partitions

**File:** `drizzle/0002_partition_trades.sql`  
**Action:** Replace the hard-coded `CREATE TABLE trades_2026_04 PARTITION OF trades …`
and `CREATE TABLE order_book_snapshots_2026_04 PARTITION OF order_book_snapshots …` blocks
with a `DO $$ … $$` PL/pgSQL block that computes the current UTC month and next month at
migration execution time, then creates the two partitions via `EXECUTE format(…)`.

**Required algorithm:**
```sql
DO $$
DECLARE
  cur_start  date := date_trunc('month', NOW() AT TIME ZONE 'UTC')::date;
  cur_end    date := (cur_start + INTERVAL '1 month')::date;
  next_start date := cur_end;
  next_end   date := (next_start + INTERVAL '1 month')::date;
  cur_tag    text := to_char(cur_start, 'YYYY_MM');
  next_tag   text := to_char(next_start, 'YYYY_MM');
BEGIN
  -- trades: current month
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS trades_%s PARTITION OF trades
     FOR VALUES FROM (%L) TO (%L)',
    cur_tag, cur_start, cur_end
  );
  -- trades: next month (ensures rows inserted at end-of-month always land somewhere)
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS trades_%s PARTITION OF trades
     FOR VALUES FROM (%L) TO (%L)',
    next_tag, next_start, next_end
  );
  -- order_book_snapshots: current month
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS order_book_snapshots_%s PARTITION OF order_book_snapshots
     FOR VALUES FROM (%L) TO (%L)',
    cur_tag, cur_start, cur_end
  );
  -- order_book_snapshots: next month
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS order_book_snapshots_%s PARTITION OF order_book_snapshots
     FOR VALUES FROM (%L) TO (%L)',
    next_tag, next_start, next_end
  );
END $$;
```

**Why two months (current + next)?**  
`PartitionManager` creates tomorrow's partition at midnight. Between midnight and the cron
running, any row with a `traded_at` in the next calendar month would have no partition.
Seeding current + next month at migration time eliminates that window.

**`IF NOT EXISTS` on partition tables:**  
Standard `CREATE TABLE … PARTITION OF` does not accept `IF NOT EXISTS` in PostgreSQL < 16.
For PostgreSQL 16+ it does. Since the stack targets PostgreSQL 16, `IF NOT EXISTS` is safe.
Alternatively, wrap in a conditional check via `pg_catalog.pg_class`. Use `IF NOT EXISTS`
(simpler) and document the PG 16 requirement.

**Preserve all other DDL in the file** (ALTER TABLE RENAME, CREATE TABLE … PARTITION BY,
indexes, INSERT … SELECT, DROP TABLE legacy). Only the hard-coded partition CREATE TABLE
statements at the bottom of each section change.

**Test:** Run the new SQL against a local PG 16 instance. Confirm:
- `\d trades` shows partitioned table
- `\d+ trades` lists two child partitions for current and next month (named `trades_YYYY_MM`)
- Same for `order_book_snapshots`
- Inserting a row with `traded_at = NOW()` succeeds
- Inserting a row with `traded_at = NOW() + INTERVAL '25 days'` succeeds when it lands in next month

### Task 3 — Register `0002_partition_trades` in `_journal.json`

**File:** `drizzle/meta/_journal.json`  
**Action:** Append a second entry to the `entries` array:

```json
{
  "idx": 1,
  "version": "7",
  "when": <unix_ms_timestamp_at_time_of_edit>,
  "tag": "0002_partition_trades",
  "breakpoints": true
}
```

**Rules:**
- `tag` must exactly match the filename stem (`0002_partition_trades`)
- `idx` must be sequential (0 = `0000_misty_thaddeus_ross`, 1 = `0002_partition_trades`)
- `when` should be a current Unix millisecond timestamp (use `Date.now()` or `date +%s000`)

**Test:** `cat drizzle/meta/_journal.json | jq '.entries | length'` → `2`

### Task 4 — Document the missing snapshot artifact

**File:** `drizzle/meta/README.md` (new file, ~15 lines)  
**Action:** Create a short note explaining why there is no `0001_snapshot.json`:

> `drizzle-kit generate` produced `0000_misty_thaddeus_ross.sql` (idx 0) and its matching
> `0000_snapshot.json`. Migration `0002_partition_trades.sql` (idx 1) is a **raw SQL
> partition conversion** that cannot be expressed in Drizzle's schema model (Drizzle does
> not support `PARTITION BY RANGE` declaratively). It was manually registered in
> `_journal.json`. No corresponding `0001_snapshot.json` is generated because Drizzle's
> introspection of a partitioned table would produce a diff against `schema.ts` and
> generate a spurious migration.
>
> **Operational rule:** Never run `drizzle-kit push` or `drizzle-kit generate` against a DB
> that has already had `0002` applied. Use `drizzle-kit migrate` (read-only against the
> journal) for all subsequent schema changes.

**Why this matters (Law's MINOR finding):** Without this note, a future developer running
`drizzle-kit generate` after `0002` is applied would see schema drift, generate a new
migration that undoes the partitioning, and break production. The note prevents this.

**Test:** File exists, readable, accurate.

### Task 6 — Add `db:migrate:partitions` fallback script (conditional)

**Triggered by:** Task 5 revealing that `drizzle-kit migrate` cannot execute raw PL/pgSQL
`DO $$ … $$` blocks (some versions of `drizzle-orm`'s migrator split on `-->
statement-breakpoint` and may choke on block-level SQL).

**If needed — `package.json`:**
```json
"db:migrate:partitions": "psql $DATABASE_URL -f drizzle/0002_partition_trades.sql"
```

**Constraint:** No new npm packages. `psql` is available in the Docker environment.  
**Documentation:** If this script is added, update `drizzle/meta/README.md` to note that
`pnpm db:migrate:partitions` must be run separately after `pnpm db:migrate`.

**Test:** `pnpm db:migrate:partitions` exits 0 against a DB that already has `0001` applied
but not `0002`. Running it twice must not fail (partition `CREATE TABLE IF NOT EXISTS`
handles idempotency).
